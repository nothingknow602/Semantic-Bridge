The Context-First Method: How to Cut ERP Reporting Migration Time by 70%

Semantic Bridge AI ~12 min read
70%
Reduction in reporting migration time 3+ weeks per Tableau workbook → under 1 week. No new headcount. No consultant. Here is how.
Every ERP transformation eventually hits the same wall. The go-live plan has a line item for data migration. For system integration testing. For training. What it almost never has — not really, not with enough budget or enough people — is a line item for reporting continuity.

And then month three hits. Finance can't close. Sales can't see pipeline. Operations is running on Excel exports from a system you're trying to retire. And your BI team is staring at a backlog of broken dashboards with no clear path through.

This is not a technology problem. It is a context problem.

Here is the method we use to solve it.

The setup: two buckets, one starting point

When an ERP transformation impacts your existing Tableau dashboards, every affected workbook falls into one of two buckets.

Fig. 1 — Decision framework

In both cases, the starting point is identical: the backend SQL query powering the Tableau data extract. And in both cases, the bottleneck is identical: before anyone can write a single line of SQL, they need to understand the old data model, the new Fusion structure, the field mappings between them, and the business logic the dashboard was built to serve.

That context-gathering process — done manually — is what was eating three-plus weeks per workbook. We stopped doing it manually.

The context-first method

The insight is simple: the reason AI-assisted SQL generation produces garbage output for most teams is not that the AI is bad. It is that the context going in is thin.

The core problem "Write me a SQL query for Fusion AP invoices" gives you nothing useful. The model does not know your chart of accounts structure, your custom flex fields, your business unit hierarchy, your DFF configurations, or which Snowflake views your Tableau workbooks actually hit.

The teams getting real results are building a rich context package before they write a single prompt. Then they use that context to generate a precise, 1-shot prompt for SQL generation — not a vague instruction, but a fully loaded specification.

Phase 1
Context harvesting

We use Microsoft Copilot's native Tableau integration as the starting point. For a given workbook, Copilot pulls five layers of context automatically:

Fig. 2 — The 5-layer context stack

The output of Phase 1 is not a document. It is a live Copilot session with enough synthesized context to understand not just the technical structure of the workbook, but its business purpose — what it is for, who depends on it, and what the data is supposed to mean.

Phase 2
Prompt generation

Here is where it gets interesting. We do not write the Snowflake AI prompt ourselves. We ask Copilot to write it.

The instruction to Copilot is roughly: "Based on everything you know about this workbook, produce a 1-shot prompt for Snowflake AI. Generate a UNION block for each view — one at a time — mapping the existing EBS structure to the new Fusion equivalent."

Copilot generates a structured prompt with five components:

Role definition You are a Snowflake SQL generator working on [view name]...
Target object Existing view structure + current EBS-sourced query logic
Task specification Generate a UNION block piping Fusion data with same output pattern as EBS query. WHERE clause logic will differ as follows...
Synthesized context Business purpose, data patterns, DFF dependencies, translation rules — distilled from all 5 context layers
Test cases ← this is the key Expected outputs, boundary conditions, known business scenarios to validate against. Copilot generates these automatically.
Why the test cases matter Most practitioners would not think to ask Copilot to generate test cases. It is one of the highest-leverage outputs of the entire method — it hands the developer a validation framework alongside the SQL scaffold, eliminating a significant portion of back-and-forth before a single line is refined.
Phase 3
SQL scaffolding

The Copilot-generated prompt goes into Snowflake AI. Snowflake AI runs a compilation check automatically and executes the generated test cases against available data. This catches syntax errors and obvious data issues before the developer touches the output.

What the developer receives is not a starting-from-scratch exercise. It is a working scaffold — SQL that compiles, that has passed basic validation, and that comes with a test case framework already defined.

Their job is refinement, not authorship.

Phase 4
Developer validation

Refinement is not just "does it run." It is scenario-based business validation. The developer takes a known business scenario — a specific order code converted from EBS to Fusion, with known order lines and totals — subsets the WHERE clause to that scenario, and verifies the output matches expectation.

Real data. Real business logic. Real validation.

Where it breaks: the honest 30%

This method does not eliminate the hard problems. It compresses the work that was previously consuming most of the time. What remains is genuinely difficult.

Fig. 4 — Failure mode map

Missing or immature test data

If you are too early in the implementation and Fusion source data is not yet populated, converted, or stable, you cannot validate the scaffold against real scenarios. The SQL may be technically correct but you have no way to confirm it.

↳ Fix: time-box the work, park it, revisit when data matures. Fighting this with more prompting is wasted effort.

Unclear translation rules

ERP transformations rarely move systems without also changing business logic. When current-state and future-state reporting logic diverge in ways nobody has documented yet, you cannot generate accurate SQL from ambiguous requirements.

↳ Fix: require an explicit current→future state mapping before SQL generation proceeds. Surface as a delivery risk.

Underfunded analytics function

In most programs, the dedicated analyst who bridges business logic and reporting logic does not exist or is spread too thin. The reporting team has to go fetch that context themselves — adding weeks of dependency on overloaded SMEs.

↳ Fix: this is a governance problem, not a technical one. Surface to program leadership before it hits the critical path.

What this becomes

What we have described here is a manually orchestrated workflow. It works. It cuts turnaround from three-plus weeks to under one week per workbook. It does not require new headcount or a consultant.

But it is still manual. The JIRA backlog, the Tableau metadata, the email threads, the meeting transcripts, the translation tables — they live in separate systems that do not talk to each other.

JIRA work items triggering context harvesting automatically
Tableau metadata feeding directly into prompt construction
Translation tables maintained as a living semantic layer
Snowflake AI integrated into the pipeline — not copy-pasted

That is what this is building toward. For now, the manual method is enough to demonstrate what is possible when context — not code — is treated as the primary input.
